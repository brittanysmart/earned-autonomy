# Working preferences

- Be concise. Explain *why*, not just *what* — especially for scoring logic, governance
  rules, and any confidence-threshold decisions.
- Governance thesis for this whole project: agents propose, humans dispose. Nothing should
  auto-execute or auto-publish. Every generated flag/action needs a visible confidence score
  and an approve/reject/edit path. Keep this contrast legible in the UI, not just the data.
- `audit.py` scoring calls a real local model (Ollama + LiteLLM, tool-calling), not regex —
  decided deliberately (GitHub issue #2) after the regex version's confidence numbers turned
  out to trace back to hand-authored trigger phrases, not real reasoning. The model rates its
  own confidence; `requires_human_review = confidence < 90` stays a hardcoded policy constant
  in code, untouched by the model — that split is the actual governance mechanism now, not
  "every number traces back to one visible regex line."

## Stack conventions

- Frontend: Next.js + shadcn/ui, TypeScript strict mode. Match shadcn's own conventions
  (component structure, `cn()` utility, Tailwind class ordering) rather than inventing new
  patterns.
- Backend/scripts: Python 3, stdlib-first by default. `audit.py` also depends on `litellm`
  (pinned in `requirements.txt`, installed into `.venv/`) — a deliberate exception: it's the
  OpenAI-compatible interface in front of the local Ollama model, not scope creep.
- `output/` is generated by `audit.py` and gitignored — don't hand-edit `flags.json`.

## Code review

- When running `/code-review` on a diff, also name-check the changed code against Fowler's
  refactoring smells and call out any that apply by name: mysterious name, duplicated code,
  long function, feature envy, data clumps, primitive obsession, repeated switches, divergent
  change, shotgun surgery, speculative generality, message chains, middleman. Naming the smell
  explicitly (not just "this could be cleaner") gets a concrete, actionable fix. Cheap to do,
  and it catches quality issues a correctness-only pass slides past.
- This augments the correctness review, it doesn't replace it — a real bug still outranks a
  smell.

## Commit message style

- Before drafting any commit message, re-read CONTEXT.md. The message and any judgment calls
  about what to highlight should reflect this project's actual purpose and intent — not just
  describe the diff mechanically.
- First line: short, plain-English summary of what changed (imperative mood: "Add," "Fix,"
  "Move"). Plain English, not jargon-dense — but still technically precise, not dumbed down.
- Body (if needed): explain WHY the change was made, not just what — especially any tradeoff
  or design decision. Write it like explaining to a competent developer who wasn't in the
  room, not to a non-technical person.
- No filler, no "this commit does X, Y, and Z" bullet-listing every file touched — one clear
  reason is better than an exhaustive list.
- Format: blank line between subject and body, blank line before the co-author trailer.
  Subject line under ~72 chars, no period at the end. Body wrapped at ~72 chars per line.
- Co-Authored-By line must use angle brackets around the email — `Co-Authored-By: Claude
  Sonnet 5 <noreply@anthropic.com>` — not bare, or GitHub/git won't parse it as a proper
  trailer.
- Keep the Claude Code co-authorship trailer as-is unless the commit is substantially my own
  hand-edit after Claude's draft.
- Always show me the drafted commit message and wait for my confirmation before running
  `git commit`. Never commit automatically without me seeing and approving the message first.

## General working style

- I like drafts before anything gets sent, posted, or committed — show me first, don't act
  automatically on multi-step or externally-visible actions.
